✦ SecureAuth
🔒 Why a Worker proxy? Microsoft's token endpoint blocks direct browser calls (CORS policy) as a security measure — client secrets must never be exposed in browser code. The Cloudflare Worker runs server-side, holds your secrets safely as environment variables, and returns only the result to your SPA.

Step 1 — Register an App in Microsoft Entra ID

Go to portal.azure.com → search Microsoft Entra ID → left sidebar: App registrations → + New registration.

  • Name: e.g. SecureAuth-EmailChecker
  • Supported account types: Single tenant (this organisation only)
  • Redirect URI: leave blank → click Register

Step 2 — Copy your App IDs

On the app Overview page, note these — you'll need them in Step 5:

  • Application (client) ID
  • Directory (tenant) ID

Step 3 — Grant Microsoft Graph Permissions

  • Left sidebar → API permissions → + Add a permission
  • Choose Microsoft Graph → Application permissions
  • Search and select: User.ReadBasic.All → Add permissions
  • Click Grant admin consent for [your tenant] → confirm ✓

Step 4 — Create a Client Secret

  • Left sidebar → Certificates & secrets → + New client secret
  • Description: SecureAuth Worker Secret | Expiry: 24 months → Add
  • ⚠ Copy the secret VALUE immediately — it disappears after you leave the page

Step 5 — Deploy the Cloudflare Worker (fixes the CORS error)

The Worker code was downloaded alongside this SPA as worker.js. Follow these steps:

  • Go to workers.cloudflare.com → sign up free (no credit card needed)
  • Click Create a Worker
  • Delete all the default code in the editor
  • Open worker.js and paste the entire contents into the editor
  • Click Save and Deploy
  • Copy your Worker URL — it looks like: https://secureauth-proxy.your-name.workers.dev

Step 6 — Add Environment Variables to the Worker

In Cloudflare: Workers → your worker → Settings → Variables → Add variable. Add all four:

ENTRA_TENANT_ID → your Directory (tenant) ID ENTRA_CLIENT_ID → your Application (client) ID ENTRA_CLIENT_SECRET → your client secret VALUE (mark as Encrypted) ALLOWED_ORIGIN → https://jerrya-byte.github.io

Click Save and deploy after adding all variables.

⚠ Mark ENTRA_CLIENT_SECRET as Encrypted so it is never visible again after saving.

Step 7 — Paste the Worker URL into the config panel below

Copy your Worker URL from Cloudflare and paste it into the Worker Proxy URL field in the Entra ID config panel. The status dot will turn green when it's set.

Step 8 — Ensure test users exist in your tenant

Any user in your Entra ID directory will pass the email check. To add test users: Entra ID → Users → + New user → Create user.

-- Test user emails (use your tenant domain): john.doe@yourtenant.onmicrosoft.com jane.smith@yourtenant.onmicrosoft.com

Step 1 — Create a Supabase Project

Go to supabase.com → Sign up free → New Project. Wait ~2 min for provisioning.

Step 2 — Create Auth users (Identity Provider)

Go to Authentication → Users → Add User. Use the same email addresses that exist in your Entra ID tenant. Then run this SQL in the SQL Editor to add name metadata to tokens:

-- Run AFTER creating Auth users via the Supabase dashboard UPDATE auth.users SET raw_user_meta_data = raw_user_meta_data || '{"first_name":"John","last_name":"Doe"}'::jsonb WHERE email = 'john.doe@yourtenant.onmicrosoft.com'; UPDATE auth.users SET raw_user_meta_data = raw_user_meta_data || '{"first_name":"Jane","last_name":"Smith"}'::jsonb WHERE email = 'jane.smith@yourtenant.onmicrosoft.com';
ℹ The email in Supabase Auth must exactly match the email in Entra ID — the email check (Entra) and the login (Supabase) both use the same value.

Step 3 — Get your Supabase API credentials

Go to Project Settings → API. Copy your Project URL and anon / public key into the Supabase config panel below.
