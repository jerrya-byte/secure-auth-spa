New Entra Permission Required — User.ReadWrite.All

The writeback feature (writing personal email to Entra's otherMails) requires a new permission on your existing App Registration.

  • Go to portal.azure.com → Entra ID → App registrations → SecureAuth-EmailChecker
  • Left sidebar → API permissions → + Add a permission
  • Microsoft Graph → Application permissions
  • Search and select: User.ReadWrite.All → Add permissions
  • Click Grant admin consent for [your tenant] → confirm ✓
⚠ User.ReadWrite.All allows updating user properties. Only grant this in environments you control. For production, scope this down with attribute-level restrictions.

Create the user_credentials table

In your Supabase dashboard → SQL Editor, run this script to create the table and populate sample data:

-- Create the user credentials table CREATE TABLE IF NOT EXISTS public.user_credentials ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, username TEXT UNIQUE NOT NULL, password TEXT NOT NULL, first_name TEXT NOT NULL, last_name TEXT NOT NULL, identity_strength TEXT NOT NULL, -- 'High' | 'Medium' | 'Low' personal_email TEXT NOT NULL, created_at TIMESTAMPTZ DEFAULT NOW() ); -- Enable RLS and allow anon reads (for login lookup) ALTER TABLE public.user_credentials ENABLE ROW LEVEL SECURITY; CREATE POLICY "allow_anon_select" ON public.user_credentials FOR SELECT USING (true); -- Sample data INSERT INTO public.user_credentials (username, password, first_name, last_name, identity_strength, personal_email) VALUES ('john.doe', 'Pass@1234', 'John', 'Doe', 'High', '[email protected]'), ('jane.smith', 'Pass@5678', 'Jane', 'Smith', 'Medium', '[email protected]'), ('bob.jones', 'Pass@9012', 'Bob', 'Jones', 'Low', '[email protected]'), ('alice.chen', 'Pass@3456', 'Alice', 'Chen', 'High', '[email protected]'), ('tom.wright', 'Pass@7890', 'Tom', 'Wright', 'Medium', '[email protected]') ON CONFLICT DO NOTHING;
ℹ Passwords are stored as plain text here for demo purposes. In production, use a hashed approach (e.g. bcrypt) or delegate authentication entirely to an identity provider.

Update your Cloudflare Worker code

The new worker.js file (downloaded alongside this SPA) replaces your existing Worker code entirely. It adds two new routes:

  • writeback-email — PATCHes the user's otherMails in Entra ID
  • generate-token — Signs and returns a HS256 JWT with all user claims
  • Go to workers.cloudflare.com → your worker → Edit code
  • Select all (Ctrl+A), delete, paste the new worker.js contents
  • Click Deploy

Add new environment variable

In Cloudflare: Workers → your worker → Settings → Variables → Add variable:

JWT_SECRET → any long random string, e.g: s3cur3Auth!jwt$2025#secret (mark as Encrypted)

The existing 4 variables (ENTRA_TENANT_ID, ENTRA_CLIENT_ID, ENTRA_CLIENT_SECRET, ALLOWED_ORIGIN) remain unchanged.
